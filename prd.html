HADR Monitor — Product Requirements

Version 0.1 (draft) 8 July 2026 Repo: hadr-starter Status: for review before Slice 1

An unattended agent that watches three live disaster feeds — GDACS, USGS and ReliefWeb — filters out the noise, assesses what remains, and publishes a morning situation report to dashboard.html at 08:30 Singapore time. When nothing has changed, it stays quiet.

1. Problem & context

A reader who cares about humanitarian disasters faces two failure modes: drowning (the USGS feed alone carries ~200 events a day, almost all irrelevant) and missing the one that matters (a shallow M6.0 under a vulnerable city, reported at 3 a.m.). The three public sources that cover this space are different kinds of systems — USGS is a sensor catalog (minutes latency, earthquakes only), GDACS is an impact model (Green/Orange/Red predictions of humanitarian need, ~25 min–hours), and ReliefWeb is a curated editorial archive (hours–days) — and none of them is append-only: events are revised, upgraded, merged and deleted after first publication.

The product turns those three sources into one trustworthy morning read: what happened, where, how bad, who is affected, and what changed since yesterday's report.

2. Goals and non-goals

Goals

  • One situation report per day at 08:30 SGT, published to dashboard.html, generated unattended.
  • Triage by humanitarian impact, not physical magnitude.
  • Honest change tracking: upgrades, downgrades and retractions are reported, never silently rewritten.
  • Silence is meaningful: no report update means "nothing significant changed", never "the pipeline broke".
  • Every statement on the dashboard traces to a machine-recorded fact.

Non-goals

  • Not a real-time alerting system. Daily cadence; no pager duty, no push notifications.
  • Not a conflict monitor. GDACS covers seven natural hazards; USGS covers one. Conflict and displacement crises are out of scope and the dashboard says so.
  • Not an authority. The report links to sources and never substitutes for national warning systems (per GDACS terms of use).
  • No tsunami warnings. The USGS tsunami flag does not indicate an actual tsunami; issuing tsunami warnings requires tsunami.gov, which is out of scope for v1.

3. Primary user & scenario

A single analyst opens dashboard.html with coffee at 08:30 SGT. In under two minutes they know: (a) any new Orange/Red-class incidents overnight, (b) status changes to incidents already being tracked — escalations first, (c) corrections/retractions, and (d) whether any data source was unreachable, so absence of news is trustworthy.

5. System overview

GitHub Actions cron — 00:20 UTC (≈ 08:30 SGT after jitter; SG has no DST)
│
├─ 1 FETCH        one fetcher per source; failures isolated per source
├─ 2 NORMALIZE    source payloads → common event schema (UTC internally)
├─ 3 RECONCILE    diff vs data/state.json → NEW / UPGRADED / DOWNGRADED /
│                 REVISED / NEW_EPISODE / DELETED
├─ 4 CORRELATE    GLIDE → GDACS↔USGS id bridge → fuzzy (type+country+time+distance)
├─ 5 TRIAGE       impact rules (§6) → significant-change set + facts.json
├─ 6 GATE         deterministic script: empty set → exit quiet, no model call
│      └─ only if changed:
├─ 7 ASSESS       headless Claude runs /sitrep skill on facts.json only
└─ 8 PUBLISH      commit dashboard.html + data/state.json to the repo
The model boundary (structural rule, from sitrep.yml) Deterministic code fetches, diffs, correlates and decides whether anything changed. The model writes prose from a facts file. The model never fetches a feed, never decides whether to wake up, and never states a fact absent from facts.json.

State lives in the repo (data/state.json, committed each run): per-source event records with revision history, the alias/correlation table, and a reporting ledger — which incidents were reported at which alert level. "Changed" always means changed relative to what the reader was last told, not relative to the last poll. Git history is the audit log.

6. Triage policy

SignalRuleRationale
GDACS alert level Report Orange and Red (event-level alertlevel, the running max). Green is counted, not itemised. Encodes predicted need for international assistance — the product's definition of "matters".
USGS PAGER Report alert ∈ {yellow, orange, red}. Magnitude alone never triggers a report. Magnitude is a poor impact proxy (M7 offshore ≪ shallow M6 under a city).
ReliefWeb disaster record Any new disaster record is report-worthy. A human editor at UN OCHA already decided it matters — free triage.
Level changes Upgrades lead the report; downgrades and retractions appear in "changes since yesterday". Honest change tracking; a yellow→red upgrade is the event the reader most needs.
Droughts (GDACS DR) Exempt from daily "new" logic; one weekly status line. Slow-onset: 10-day update cycle, months-long lifetime; daily diffing produces noise.
Cross-hazard ranking Never sort by alertscore across hazard types. Scales differ per hazard (EQ continuous; TC/FL/VO snap to 0.5/1.5/2.5; DR expert scale).

7. Functional requirements

Fetch & normalize

IDRequirementAcceptance
FR-1P0 Fetch USGS via FDSN query with updatedafter cursor from state — returns new and revised events in one call; immune to rolling-window edges on delayed cron runs. A magnitude revision to a 3-day-old event appears in the next run's diff.
FR-2P0 Fetch GDACS EVENTS4APP; per-event detail and geometry fetched lazily, Orange+ only. Green events cause no detail requests.
FR-3P1 Fetch ReliefWeb disasters via RSS; GLIDE parsed from item description. API v2 implementation behind the same interface, activated when appname approved. Swapping RSS→API changes no code outside the fetcher module.
FR-4P0 A source failure never fails the run: fetcher returns data or a typed failure recorded in state. With GDACS unreachable, the run completes and the report carries a staleness banner.
FR-5P0 Normalize to a common schema: (source, stable_key, hazard, occurred_at, geo, impact_signal, raw); all times UTC internally, SGT only at render. USGS ms-epoch and GDACS local-naive timestamps compare correctly.

Reconcile & correlate

IDRequirementAcceptance
FR-6P0 USGS identity by ids alias-set intersection, never by single id; alias sets merged on every run. An event whose preferred id flips ci…→us… is not duplicated.
FR-7P0 GDACS identity by eventtype+eventid; episodes stored as versions; event-level and episode-level alert levels both tracked. A new episode raising Orange→Red emits UPGRADED, not NEW.
FR-8P1 Detect deletions: previously reported USGS events re-checked with includedeleted=true; deletions emit a retraction fact. A deleted event we reported yesterday produces a retraction line, not silence.
FR-9P1 Correlate into incidents by priority: GLIDE (late-binding, re-checked every run) → GDACS↔USGS id bridge for earthquakes → fuzzy (hazard + country + ±48 h + <300 km). The same quake from USGS and GDACS renders as one incident with two source views.

Triage, gate, report

IDRequirementAcceptance
FR-10P0 Apply §6 triage to the reconciled diff; output facts.json: incidents, typed changes vs the reporting ledger, staleness warnings. Every dashboard statement maps to a facts.json field.
FR-11P0 scripts/check_changes.py: deterministic, no model call, exit status the workflow branches on. Empty change set → workflow ends with no model invocation and no dashboard commit.
FR-12P0 The /sitrep skill (headless claude -p) writes dashboard.html from facts.json only: lead incidents, "changes since yesterday" (upgrades, downgrades, retractions), staleness banner, source links. Report contains no claim absent from facts.json; retractions phrased as corrections.
FR-13P0 Publish: workflow commits dashboard.html + data/state.json; state committed even on quiet runs (proof of life). A quiet day still shows a state commit; the dashboard timestamp is unchanged.
FR-14P2 Raw payload snapshots archived per run for replay/backtesting of triage rules. Yesterday's run reproducible from snapshots without network access.

8. Non-functional requirements

  • NFR-1 Politeness: one fetch per source per run; descriptive User-Agent; registered appname on every ReliefWeb API call; back off and degrade to RSS on 4xx — never hammer-retry.
  • NFR-2 Schedule: cron at 00:20 UTC (Actions cron jitters at popular minutes; Singapore has no DST, so a fixed UTC time is correct year-round).
  • NFR-3 Trust: silence means "no change", so pipeline failures must be loud — a failed run notifies (Actions failure email/badge), never fails silently.
  • NFR-4 Runtime: Python (repo standard; local interpreter is 3.9 — pin CI to match or upgrade local first). No servers, no databases: state is a committed JSON file.
  • NFR-5 Licensing: GDACS content attributed CC BY 4.0; ReliefWeb items linked with source attribution, full document text never republished; USGS public domain.

11. Open decisions

  • Geographic scope: global, or weighted to a region of interest? (Affects triage thresholds; default: global.)
  • Dashboard on quiet days: untouched file vs. a "no significant change as of <date>" stamp. (Default: untouched, per "stays quiet".)
  • Snapshot retention: how many days of raw payloads to keep in-repo before pruning. (Default: 14.)
  • Whether Slice 5 adds tsunami.gov as a fourth source or defers past the course.

12. Definition of done (Wednesday)

  • Scheduled workflow runs unattended and has produced at least one real morning report.
  • A quiet run provably makes no model call and no dashboard change.
  • One incident on the dashboard is correlated from ≥2 sources.
  • A revision (level change or retraction) has flowed through to a "changes since yesterday" entry.
  • Artefacts present: prd.html, system-view.html, implementation-notes.md, dashboard.html, goal.md, the /sitrep skill.
